# Спецификация: Sudoku PWA

## 1. Обзор проекта

Судоку в браузере в виде Progressive Web App (PWA), устанавливаемое на Android и iOS (телефон/планшет) через "Добавить на экран" / нативный install-prompt браузера. Полностью работает офлайн после первой загрузки.

**Цель:** pet-проект / портфолио. Публикация в сторы (App Store / Google Play) не планируется на старте, но архитектура должна допускать обёртку в Capacitor в будущем без переписывания логики.

## 2. Технологический стек

| Слой | Технология |
|---|---|
| Язык / сборка | TypeScript + Vite |
| UI | React 18 |
| Стили | CSS Modules или Tailwind (на выбор агента, зафиксировать в начале) |
| Состояние | React Context + useReducer (достаточно, Redux избыточен для этого объёма) |
| PWA | `vite-plugin-pwa` (генерация manifest.json + service worker на базе Workbox) |
| Хранение данных | `localStorage` для настроек и текущей игры; `IndexedDB` (через `idb`) для истории игр/статистики, если объём вырастет |
| Тесты | Vitest (unit-тесты для генератора/решателя судоку — это критическая логика) |
| Линтинг | ESLint + Prettier |

## 3. Функциональность MVP (средний объём)

### 3.1 Игровой процесс
- Генерация судоку с гарантией единственного решения.
- 3 уровня сложности: Лёгкий / Средний / Сложный (регулируется количеством открытых клеток + техниками, требуемыми для решения без подбора).
- Ввод цифр: тап по клетке → тап по цифре в панели ввода (не системная клавиатура).
- Режим заметок (pencil marks) — маленькие цифры-кандидаты в клетке, переключается отдельной кнопкой-тумблером.
- Подсветка: одинаковые цифры на поле, текущие строка/столбец/блок 3×3, конфликтующие (ошибочные) значения.
- Таймер игры (пауза при уходе из приложения / сворачивании).
- Undo (минимум 1 шаг назад, желательно полная история хода в рамках партии).
- Проверка завершения (детект победы, финальный экран с временем).
- Сохранение текущей незавершённой партии — при закрытии приложения прогресс не теряется (сохранять в `localStorage` при каждом ходе или с debounce).

### 3.2 Статистика по играм
- Лучшее время прохождения по каждой сложности (Лёгкий/Средний/Сложный).
- Количество завершённых партий по каждой сложности.
- Текущий streak (подряд идущие завершённые партии без выхода без сохранения/сброса — уточнить точную логику при реализации).
- Хранится отдельно от текущей партии (см. раздел 6, `StatsState`), обновляется при переходе партии в статус `completed`.
- Экран/блок статистики — простой список значений, без графиков на MVP.

### 3.3 Не входит в MVP (вынести в раздел 10 "На будущее")
- Темы оформления, подсказки (hints), несколько сохранённых партий одновременно, звук/анимации, мультиплеер.

## 4. PWA-специфика

### 4.1 Manifest (`manifest.json`, генерируется через vite-plugin-pwa)
- `name`, `short_name`, `description`
- `display: "standalone"`
- `orientation`: не фиксировать жёстко — приложение должно адаптироваться и под portrait, и под landscape (важно для планшетов)
- `theme_color`, `background_color`
- Иконки: минимум 192×192 и 512×512 (`purpose: "any maskable"` — обязательно maskable-версия для Android, иначе иконка обрежется в круг с артефактами)

### 4.2 iOS-специфика (это самое узкое место, закладываем сразу)
- iOS **не поддерживает** `beforeinstallprompt` — нет автоматического баннера установки. Нужен собственный UI-блок с инструкцией: "Нажмите Поделиться → На экран Домой" (показывать один раз, определяя iOS Safari через `navigator.userAgent` / feature detection, скрывать после первого показа через флаг в localStorage).
- Обязательные мета-теги в `index.html`:
  ```html
  <meta name="apple-mobile-web-app-capable" content="yes">
  <meta name="apple-mobile-web-app-status-bar-style" content="black-translucent">
  <link rel="apple-touch-icon" href="/icons/apple-touch-icon.png">
  <link rel="apple-touch-startup-image" href="...">
  ```
- Safari на iOS может **очищать данные Service Worker/Cache Storage** после ~7 дней неактивности сайта (ITP policy) — учесть, что офлайн-кэш может слететь у редко заходящих пользователей. `localStorage` с текущей партией менее подвержен, но тоже не гарантирован вечно — не полагаться на него как на единственное хранилище для чего-то важного.
- Push-уведомления не закладываем в MVP — на iOS они требуют 16.4+ и добавленное на экран приложение, это отдельная задача при необходимости.

### 4.3 Service Worker / кэширование
- Стратегия: `CacheFirst` для статики (JS/CSS/шрифты/иконки), `NetworkFirst` или просто отсутствие сети (приложение не ходит в сеть за данными вообще — судоку генерируется на клиенте, бэкенда нет).
- Precache всех ассетов при установке SW — приложение должно полностью открываться и работать без сети сразу после первого визита.

## 5. Алгоритм судоку

### 5.1 Генерация
1. Сгенерировать полностью решённое поле 9×9 (backtracking + случайная перестановка порядка чисел на каждом шаге для рандомизации).
2. Удалять числа по одному в случайном порядке; после каждого удаления проверять через solver, что решение остаётся **единственным**.
3. Останавливать удаление по достижении целевого количества открытых клеток для выбранной сложности (примерные ориентиры: Лёгкий ~36–40 открытых, Средний ~30–35, Сложный ~24–28 — точные числа откалибровать тестами, для сложных дополнительно проверять, что задача не решается тривиальным перебором "одна возможная цифра в клетке", а требует более продвинутых техник).

### 5.2 Solver (нужен и для генерации, и для рантайм-проверки конфликтов)
- Backtracking-решатель, возвращающий (а) решаемо ли поле, (б) единственно ли решение (для этого — не останавливаться на первом найденном решении, а продолжать поиск до второго или до исчерпания).
- Отдельная быстрая функция валидации хода (проверка строки/столбца/блока) для подсветки конфликтов в реальном времени — не гонять полный solver на каждый клик.

### 5.3 Тестирование этой логики — обязательно
Это единственная часть приложения, где баг критичен (нерешаемая головоломка = сломанный продукт). Vitest-тесты:
- Генератор всегда даёт валидное и единственно решаемое поле (прогнать генерацию N раз в тесте).
- Solver корректно определяет решаемость на known-answer кейсах.
- Валидатор конфликтов корректно ловит дубли в строке/столбце/блоке.

## 6. Модель данных (localStorage)

```ts
interface GameState {
  puzzleId: string;
  difficulty: 'easy' | 'medium' | 'hard';
  initialGrid: number[][];      // 0 = пустая клетка
  currentGrid: number[][];
  solution: number[][];
  notes: Set<number>[][];       // заметки-кандидаты по клеткам
  history: Move[];              // для undo
  elapsedSeconds: number;
  startedAt: string;            // ISO
  status: 'in_progress' | 'completed';
}

interface Move {
  row: number;
  col: number;
  prevValue: number;
  newValue: number;
  wasNote: boolean;
}

interface StatsState {
  byDifficulty: Record<'easy' | 'medium' | 'hard', {
    bestTimeSeconds: number | null;
    completedCount: number;
  }>;
  currentStreak: number;
}
```

## 8. UI/UX и адаптивность

- Один основной экран (без роутинга на MVP): поле + панель ввода + хедер с таймером/кнопками.
- Адаптивная сетка: на телефоне поле на всю ширину экрана, панель ввода снизу; на планшете и в landscape — поле и панель ввода бок о бок, используя лишнее пространство.
- Touch-friendly размеры тапабельных зон (минимум 44×44px, ориентир Apple HIG).
- Тестировать на реальных пропорциях: iPhone SE (маленький), iPhone Pro Max, iPad — минимум через DevTools device emulation, если нет физических устройств.

## 9. Этапы разработки (для агента, рекомендуемый порядок)

1. **База проекта**: Vite + React + TS, настройка ESLint/Prettier, базовый `vite-plugin-pwa` конфиг с плейсхолдер-иконками.
2. **Core-логика**: generator, solver, validator + Vitest-тесты. Это можно и нужно сделать до UI — логика проверяется в консоли/тестах.
3. **Игровое поле**: рендер сетки, ввод чисел, подсветка, детект победы — без сохранения и таймера.
4. **Состояние и сохранение**: GameContext, localStorage-персистентность, восстановление партии при перезапуске.
5. **Доп. механики**: заметки, undo, таймер с паузой, выбор сложности, экран новой игры.
6. **Статистика**: `StatsState`, обновление при завершении партии, экран/блок отображения.
7. **PWA-полировка**: финальные иконки (обычные + maskable + apple-touch), manifest, кастомный InstallPrompt-компонент с детектом iOS/Android, service worker precache, проверка офлайн-режима (DevTools → offline).
8. **Ручное тестирование на устройствах**: установка на реальный/эмулируемый Android и iPhone, оба размера экрана.

## 10. Критерии приёмки (Definition of Done)

- [ ] Приложение устанавливается на Android (появляется install-prompt или доступно через меню браузера) и на iOS (через "Добавить на экран" с понятной инструкцией внутри приложения).
- [ ] После установки полностью работает в офлайн-режиме (проверено отключением сети).
- [ ] Судоку любой сложности решаемо и имеет единственное решение (покрыто тестами).
- [ ] Прогресс партии не теряется при закрытии/перезапуске приложения.
- [ ] Корректно отображается на телефоне и планшете, в portrait и landscape.
- [ ] Undo, заметки, таймер, смена сложности работают без багов.
- [ ] Статистика (лучшее время, количество побед, streak) корректно обновляется при завершении партии и сохраняется между запусками.
