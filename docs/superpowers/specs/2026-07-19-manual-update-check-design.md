# Кнопка «Проверить обновления» в настройках

**Дата:** 2026-07-19
**Статус:** дизайн утверждён, реализация не начата

## Цель

Дать игроку способ **вручную** спросить сервер о новой версии приложения.

Сегодня строка «Версия» в настройках содержит кнопку «Обновить», задизейбленную
до тех пор, пока service worker сам не обнаружит обновление (`needRefresh`).
Игрок видит серую кнопку и не может ничего предпринять — принудительно запросить
`sw.js` невозможно. Особенно заметно в standalone-режиме, который редко закрывают
полностью, из-за чего фоновая проверка откладывается.

Кнопка не заменяет автоматический механизм из [ADR-0004](../../adr/0004-pwa-update-strategy.md),
а добавляет **второй вход** в то же состояние «обновление готово».

## Ключевые решения

| Вопрос | Решение | Почему |
|---|---|---|
| Сколько кнопок в строке «Версия» | Одна, меняющая роль по состоянию | Строка уже плотная (версия + кнопка); состояния взаимоисключающие |
| Исходы проверки | Три различимых: офлайн / ошибка / не найдено | Сказать «у вас последняя версия» человеку без интернета — вранье |
| Как понять «обновления нет» | Окно ожидания 10 с после `update()` | `update()` не сообщает результат; точное отслеживание `installing` не окупается |
| Формулировка исхода по таймауту | «Обновлений не найдено», не «Актуальная версия» | Верно и когда нового нет, и когда не успело скачаться |
| API хука наружу | Один обработчик `handleVersionAction` | Кнопка одна — действие одно; развилка «проверить или применить» скрыта в хуке |
| Тестирование | Чистая функция переходов + тесты компонента | Таймер и офлайн проверяются без фейковых таймеров и моков SW |

**Не входит в scope:**

- **Версионирование** (`APP_VERSION` не меняется между релизами) — отдельная фича,
  заведена в роадмапе. Зависимости нет: кнопка показывает то, что лежит в `APP_VERSION`,
  независимо от того, как эта строка туда попадает.
- **Автопроверка при открытии настроек** — можно добавить поверх позже.
- **E2E-тест** — реальный SW-апдейт в Playwright дорог и хрупок.
- **Изменения в `UpdateBanner`** — продолжает работать на `updateAvailable`/`applyUpdate`.

## Состояния строки «Версия»

Состояния взаимоисключающие, поэтому моделируются одним union-типом, а не набором
булевых флагов.

| Состояние | Кнопка | Подпись | Поведение |
|---|---|---|---|
| `idle` | активна | «Проверить обновления» | запускает проверку |
| `checking` | disabled | «Проверяем…» | ждёт результата |
| `updateReady` | активна | «Обновить» | применяет обновление |
| `notFound` | активна | «Обновлений не найдено» | через 4 с → `idle` |
| `offline` | активна | «Нет соединения» | через 4 с → `idle` |
| `failed` | активна | «Не удалось проверить» | через 4 с → `idle` |

Правила переходов:

- `updateReady` — **терминальное**: обновление найдено, кнопка остаётся «Обновить».
- `updateFound` **выигрывает у любого состояния**: если `needRefresh` пришёл, пока
  показывается `notFound`, состояние переключается на `updateReady`.
- `noticeExpired` действует только на три «сообщающих» состояния; в `updateReady`
  и `checking` игнорируется.
- В `updateReady` клик применяет обновление; в `checking` кнопка недоступна;
  во всех остальных клик запускает проверку.

## Архитектура

Три единицы, каждая с одной ответственностью.

### `src/state/updateCheckState.ts` — чистая логика

Ноль зависимостей от React и service worker. Тип состояния, тип события и функция
перехода:

```ts
export type UpdateCheckState =
  | 'idle' | 'checking' | 'updateReady' | 'notFound' | 'offline' | 'failed';

export type UpdateCheckEvent =
  | 'checkStarted' | 'checkOffline' | 'checkFailed'
  | 'checkTimedOut' | 'updateFound' | 'noticeExpired';

export function nextUpdateCheckState(
  args: { current: UpdateCheckState; event: UpdateCheckEvent }
): UpdateCheckState;
```

Реализация — лукап-таблица переходов, не цепочка `if`/`switch`.

### `src/state/appUpdate.ts` — хук, расширяется

Держит состояние через `useState`, вычисляет переходы через `nextUpdateCheckState`,
владеет всеми побочными эффектами: `navigator.onLine`, `registration.update()`, таймеры.

```ts
export interface AppUpdateApi {
  updateAvailable: boolean;        // остаётся — на нём живёт UpdateBanner
  applyUpdate: () => void;         // остаётся — баннер зовёт напрямую
  checkState: UpdateCheckState;    // новое
  handleVersionAction: () => void; // новое: проверить или применить
}
```

`ServiceWorkerRegistration` нужен, чтобы вызвать `update()`. `useRegisterSW` отдаёт
его только через колбэк — кладём в `useRef`:

```ts
useRegisterSW({
  onRegisteredSW: (_swUrl, registration) => { registrationRef.current = registration; },
});
```

### `src/components/settings/SettingsScreen.tsx` — только рендер

Берёт из хука `checkState` и `handleVersionAction`. Подпись кнопки — из
`Record<UpdateCheckState, string>`. `disabled` только при `checking`.

## Поток проверки

```
клик
 │
 ├─ checkState === 'updateReady' → applyUpdate()
 │                                 (saveGame + updateServiceWorker(true))
 │
 └─ иначе
      │
      ├─ navigator.onLine === false      → offline
      ├─ registrationRef.current == null → failed
      │
      └─ checking + таймер 10 с
           │
           └─ registration.update()
                ├─ reject             → failed
                └─ resolve → ждём:
                     needRefresh стал true → updateReady
                     таймер истёк          → notFound
```

Пустой `registrationRef` означает, что SW ещё не зарегистрирован или отключён
(dev-режим) — в сеть не идём, сразу `failed`.

## Обработка ошибок и утечек

- **Таймеры в рефах, очистка в `useEffect`-cleanup.** Иначе `setState` после
  размонтирования настроек: игрок может уйти с экрана во время `checking`.
- **Повторный клик во время `checking` невозможен** — кнопка `disabled`.
- **Два экземпляра хука.** `useRegisterSW` вызывается и в `UpdateBanner`, и в
  `SettingsScreen`. Так работает и сейчас; `vite-plugin-pwa` это допускает —
  регистрация одна, подписок несколько. Логика проверки живёт в своём экземпляре
  и на баннер не влияет.

## Тестирование

**`src/state/updateCheckState.test.ts`** — все переходы чистой функции:

- каждое событие из каждого состояния;
- приоритет `updateFound` над `notFound`/`offline`/`failed`/`checking`;
- `noticeExpired` гасит три «сообщающих» состояния и игнорируется в `updateReady`.

**`src/components/settings/SettingsScreen.test.tsx`** — расширяем существующий мок
`useAppUpdate` полями `checkState` и `handleVersionAction`:

- подпись кнопки для каждого из шести состояний;
- `disabled` только при `checking`;
- клик вызывает `handleVersionAction`.

E2E не добавляем.
