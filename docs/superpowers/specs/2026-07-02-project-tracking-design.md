# Дизайн: система трекинга проекта (Markdown)

**Дата:** 2026-07-02
**Статус:** утверждён

## 1. Цель

Мета-слой для ведения проекта поверх кода: отслеживать, какие фичи сделаны, какие
запланированы, связи с документами (спеки, планы), заметки и архитектурные решения.
Задача переиспользуема — тот же подход применим к другим проектам.

Это НЕ фича самой игры. Это способ вести разработку и историю решений.

## 2. Принятые решения

| Вопрос | Решение |
|---|---|
| Где живёт | Markdown-файлы в репозитории (вариант A). Задел под синхронизацию с GitHub Issues — на будущее, сейчас не делаем. |
| Структура | Роадмап-файл для фич + отдельный ADR-журнал для архитектурных решений (A + C). |
| Формат ADR | Один файл на решение (lightweight ADR), вариант A. |
| Дизайн-док | Хранится в `docs/superpowers/specs/` для переиспользования в других проектах. |

## 3. Артефакты

### 3.1 `docs/roadmap.md`

Единая таблица-роадмап. Колонки:

| Фича | Статус | Приоритет | Зависимости | Ветка / PR | Спека / доки | План создан | Завершено | Заметки |

- **Статус:** `📋 запланировано` · `🚧 в работе` · `✅ готово` · `🧊 отложено`
- **Приоритет:** `MVP` / `backlog`
- **Зависимости:** какие фичи должны быть готовы раньше этой
- **Ветка / PR:** где велась работа (для истории)
- **Спека / доки:** ссылка на связанный документ
- **План создан:** дата появления плана фичи (ISO `YYYY-MM-DD`)
- **Завершено:** дата окончания имплементации (ISO `YYYY-MM-DD`)
- **Заметки:** короткий комментарий; № этапа из спеки (только для MVP-фич); ссылка на ADR

Каркас создаётся пустым (заголовок + закомментированная строка-пример). Наполнение
MVP-фичами и backlog — задача сессии, создающей план реализации.

### 3.2 `docs/adr/`

- `0000-template.md` — шаблон lightweight-ADR: Заголовок, Статус
  (`предложено` / `принято` / `заменено ADR-NNNN`), Дата, Контекст, Решение, Последствия.
- `README.md` — индекс ADR (таблица: №, название, статус, дата). Пустой каркас.
- Реальные записи: `NNNN-<slug>.md`, нумерация с 0001.

### 3.3 Дополнение `CLAUDE.md`

Новая секция «Ведение проекта / трекинг»:
- Где роадмап и ADR, что значит каждая колонка.
- Правило перехода статусов: начало работы → `🚧` + ветка + дата «План создан»;
  завершение → `✅` + дата «Завершено».
- Правило ADR: значимое архитектурное/сквозное решение → новый файл из шаблона +
  строка в индексе + ссылка из колонки «Заметки» роадмапа.
- Формат дат — ISO `YYYY-MM-DD`.

## 4. Что НЕ делаем сейчас (YAGNI)

- Отдельные файлы на каждую фичу (вариант B) — добавим только если конкретная фича
  реально потребует детализации.
- Синхронизация с GitHub Issues/Projects — задел, не реализация.
- Наполнение роадмапа фичами — делает соседняя сессия при создании плана.
