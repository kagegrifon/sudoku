# ADR-0002: Стратегия хранения данных

**Статус:** принято
**Дата:** 2026-07-02

## Контекст

Приложению нужно хранить: (а) текущую незавершённую партию, (б) настройки, (в) историю завершённых партий для статистики по периодам (день/неделя/месяц/всё время). Спека (§2, §6) предлагает localStorage для настроек/партии и IndexedDB (через `idb`) для истории, «если объём вырастет».

Действующие силы:
- Статистика по периодам требует хранить **каждую** завершённую партию с датой и исходом — список растёт без явной границы.
- localStorage синхронный и ограничен ~5 МБ — плохо масштабируется под растущий журнал.
- iOS ITP может очищать хранилища после ~7 дней неактивности — нельзя полагаться на кэш как на единственный источник.
- Схемы данных будут меняться между версиями приложения (см. [ADR-0004](0004-pwa-update-strategy.md)) — нужен механизм миграций.

## Решение

- **localStorage** — текущая партия (`sudoku:game`) и настройки (`sudoku:settings`). У обоих объектов поле `schemaVersion`; при несовпадении без миграции партия отбрасывается (начинается новая).
- **IndexedDB через `idb`** — журнал завершённых партий (store `games`, записи `CompletedGame`). Миграции — через `upgrade`-колбэк `idb`.
- Агрегаты статистики **не хранятся** — считаются на лету из журнала за выбранный период.

## Последствия

- Проще: растущий журнал не упирается в лимит localStorage; агрегаты всегда согласованы (нет предрассчитанных сумм — нет рассинхрона).
- Сложнее: два разных механизма хранения и две схемы версионирования; асинхронный доступ к IndexedDB в слое статистики.
- Учитываем: на iOS данные не гарантированы вечно — это осознанное ограничение MVP, бэкапа в облако нет.
