# ADR-0005: Модель навигации по экранам

**Статус:** принято
**Дата:** 2026-07-13

## Контекст

До интеграции дизайна навигация была примитивной: `AppContext` хранил
`ActiveView = 'game' | 'stats'`, вход был сразу в игру. Дизайн (9 экранов) вводит
полноценный стартовый экран, отдельный экран настроек и оверлеи (выбор сложности,
пауза, результат). Нужна явная модель переходов с историей «назад».

Силы:
- Экранов теперь четыре (`home`, `game`, `settings`, `stats`), между ними — переходы
  с запоминанием, откуда пришли (настройки открываются и с `home`, и из `game`).
- Оверлеи (пауза, победа/поражение, bottom-sheet выбора сложности) — не самостоятельные
  экраны, а состояние внутри игрового экрана.
- Хук `useAppView` уже импортируют существующие компоненты и тесты — его имя надо сохранить.

## Решение

Навигацию храним в `AppContext` на `useReducer` (в согласии с [ADR-0003](0003-state-architecture.md)).
Форма состояния:

```ts
{ screen: 'home' | 'game' | 'settings' | 'stats'; previous: Screen | null }
```

Методы:
- `navigate(to)` — переход на экран, текущий запоминается в `previous`.
- `goBack()` — возврат в `previous ?? 'home'`.

Вход в приложение — **всегда** `home`. Оверлеи (пауза, результат, выбор сложности) —
локальный state внутри `GameScreen`, не экраны. `App.tsx` выбирает компонент экрана
через lookup-мапу `Record<Screen, () => JSX.Element>` (без `if`/`switch`-цепочек).

## Последствия

- Появляется явная, тестируемая история «назад» — настройки корректно возвращаются
  туда, откуда открыты.
- Оверлеи не засоряют глобальную навигацию; их жизненный цикл привязан к игре.
- `useAppView` сохраняет имя — существующие импорты и тесты не ломаются на переименовании,
  но их надо обновить под новое API (`navigate`/`goBack` вместо `setActiveView`).
